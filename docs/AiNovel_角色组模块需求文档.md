## 一、项目背景与目标

### 1.1 现有架构中的相关概念

- **角色（Role）**：单角色定义 + 世界观下 `role_info`（姓名、背景、性格等）+ 角色关系（两两）。章节生成时通过 `role_names`、`faction_names`、`geo_names` 等传入 LLM 上下文。
- **阵营（Faction）**：组织/势力（国家、军队、帮派等），有层级、关系、领土。偏宏观归属与组织行为。
- **章节/脑洞**：通过 `role_names`、`faction_names` 参与提纲、注意事项、续写；通过 pick 接口从文本中抽取 `roles` / `factions` / `locations`。

### 1.2 产品目标

新增**角色组**管理能力，并在章节生成、选角等场景中支持引用角色组（展开为成员 + 集体设定），使「少量角色作为一个小团体」的集体戏份更一致、更易控制。

---

## 二、角色组定义与边界

### 2.1 定义

**角色组**：在小说中用于定义**少量角色的集体**及其**集体行动方式**的实体。

### 2.2 与阵营的区别

- 阵营是「组织/势力」，角色组是「这几个人作为一个小团体的共同行为、配合方式、集体目标」。
- 示例：阵营 = 「北境军」「暗影会」；角色组 = 「主角小队」「师徒三人」「本次会议中的五人」「某次行动小组」。

### 2.3 与角色关系的区别

- 角色关系是两两关系（朋友、师徒等）；角色组是多角色 + 集体行为描述（如「三人一起行动时的分工与习惯」）。

### 2.4 核心价值

在写章节或脑洞时引用「角色组」，让 LLM 知道「这一组角色会一起出现，且按某种方式配合/行动」，避免只靠零散角色名而缺少集体行为约束。

## 三、功能需求

### 3.1 功能概述

| 维度 | 说明 |
|------|------|
| 名称 | 角色组（Character Group / Role Group） |
| 归属 | 按世界观（worldview）维度管理，与角色、阵营同级 |
| 核心属性 | 名称、描述、成员（关联已有 role_info）、集体行动说明（如分工、习惯、共同目标） |
| 使用场景 | 在章节/脑洞中选用「角色组」时，带出成员列表 + 集体行动描述，作为 LLM 上下文 |

### 3.2 用户故事

- **US1**：创作者可在某世界观下创建/编辑/删除角色组，填写名称、描述、集体行动说明，并勾选成员（从该世界观下已有角色/role_info 中选择）。
- **US2**：在章节管理（提纲、续写、注意事项）或脑洞中，可选择「角色组」；选择后自动带入该组的成员角色名 + 集体行动说明，与现有 `role_names` / `faction_names` 一起参与生成。
- **US3**（可选）：从文本中抽取实体时，支持按「角色组」名称抽取（类似 pick 的 roles/factions），便于从已有章节中识别「这一组人」的出场。

## 四、数据模型

### 4.1 基础模型（Phase 1）

- **主表 `role_group`（或 `character_group`）**
  - `id`, `worldview_id`, `name`, `description`, `collective_behavior`（集体行动/配合方式说明）, `group_type`（角色组类型）, `group_status`（角色组状态）, `sort_order`, `created_at`, `updated_at` 等。类型与状态用于扩展剧情灵活性，见第六节。
- **关联表 `role_group_member`**
  - `id`, `role_group_id`, `role_info_id`（或 `role_id` 视现有模型而定）, `sort_order`。
- 与现有 `IAiNoval` 对齐：新增 `IRoleGroup`、`IRoleGroupMember`，与 `IRoleInfo`/`IRoleData`、`IFactionDefData` 同级。

### 4.2 扩展字段（见第五节）

为支持「比较复杂的小组内交互」和「比较自然的小组行动」，在基础模型上增加第五节所述的扩展字段，可在 Phase 1 后分步落地。

## 五、支持复杂小组内交互与自然小组行动的字段设计

若要呈现**比较复杂的小组内交互**以及**比较自然的小组行动**，建议在数据模型中增加以下字段（可按阶段分批实现）。

### 5.1 支持「小组内交互」的字段

| 字段建议 | 说明 | 用途 |
|----------|------|------|
| **成员在组内角色/分工** | 如：领队、智囊、执行者、调和者、情报员等；每成员一个角色标签。可在 `role_group_member` 表增加 `role_in_group`。 | 明确谁主导、谁配合，写对话与行动时有依据。 |
| **成员间关系或互动模式** | 短文本或结构化：如「A 与 B 常争执」「B 听 C 的」「C 负责打圆场」。可在成员表增加 `notes_with_others`，或在主表增加 `member_dynamics`（富文本）。 | 写「谁和谁怎么互动」更一致。 |
| **决策方式** | 枚举或短文本：多数决、领队决定、共识、某人有否决权等。主表字段 `decision_style`。 | 写「小组怎么做决定」更自然。 |
| **冲突点与默契点** | 两段文本：在哪些事上容易分歧 / 在哪些事上高度一致。主表字段 `conflict_points`、`accord_points`（或 `默契_points`）。 | 既有张力又有默契，交互更立体。 |
| **子对/小圈子（可选）** | 可选的子组：如「A 和 B 常一起行动」「C 单独和领队汇报」。可用 JSON 或单独表 `role_group_subset` 存储。 | 支持复杂小组内的亚团体、私下结盟。 |

**实现建议**：
- 成员表：增加 `role_in_group`（组内角色）、`notes_with_others`（与组内他人的关系/互动备注）。
- 主表：增加 `decision_style`、`conflict_points`、`accord_points`。
- 子对/小圈子：首版可用主表 JSON 字段或富文本描述，后续再考虑独立表。

### 5.2 支持「比较自然的小组行动」的字段

| 字段建议 | 说明 | 用途 |
|----------|------|------|
| **集体行动说明**（已有） | 扩展为富文本或分段：一般如何行动、习惯流程。对应基础模型中的 `collective_behavior`。 | 生成时直接给 LLM 的「小组人设」。 |
| **常见行动模式/剧本** | 短条列：遇事谁先发言、谁拍板、谁执行、谁收尾。主表字段 `action_pattern`（或 `action_script`）。 | 写「开会/遇险/谈判」等场景更顺。 |
| **情境-典型反应（可选）** | 简单 key-value：如「遭遇强敌 → 先由 A 评估，B 掩护，C 交涉」。主表字段 `situation_responses`（JSON 或单独表）。 | 高亮场景下的固定套路，避免写崩。 |
| **小组风格/节奏** | 短文本或标签：谨慎型/冲动型、低调/张扬、快节奏/谋定后动。主表字段 `group_style`。 | 统一小组的整体气质。 |
| **共同目标与禁忌** | 两段：本组共同目标；小组绝不会做的事。主表字段 `shared_goal`、`taboo`。 | 动机和边界清晰，行动更合理。 |
| **对外一致口径/习惯用语（可选）** | 小组共有的说法、口头禅、对外表态方式。主表字段 `group_mannerisms`。 | 对话和叙述更贴小组人设。 |

**实现建议**：
- 主表在基础字段上增加：`action_pattern`、`group_style`、`shared_goal`、`taboo`；可选 `situation_responses`（JSON）、`group_mannerisms`。
- `collective_behavior` 保持为主描述，上述字段作为结构化补充，便于在提示词中按需拼接。

## 六、角色组类型与状态（剧情灵活性）

通过为角色组定义**类型**和**状态**，可区分「固定小队」与「临时组合」、「活跃」与「已解散」等，在选角、生成与时间线中提供更细粒度的控制，增强剧情灵活性。

### 6.1 角色组类型（group_type）

类型描述该组在设定上的**性质与稳定性**，影响其在剧情中的出场方式、是否长期存在、是否可复用等。

| 类型枚举建议 | 说明 | 剧情用途 |
|--------------|------|----------|
| **固定小队** | 长期绑定、经常一起行动的小队（如主角团、探险队）。成员相对稳定。 | 多章节复用；选角时优先推荐；可关联故事线/时间线。 |
| **临时组合** | 因某次任务、事件或场景临时集结（如某次会议的参会者、一次行动的搭档）。 | 可在特定章节/事件下启用；任务结束后可置为「已解散」或「休眠」。 |
| **会议/集会** | 以会议、仪式、集会等场景为载体的角色集合。 | 场景驱动；可带时间、地点约束；适合单章或短跨度。 |
| **师徒/师徒团** | 以师徒关系为核心的小团体。 | 关系驱动；可与角色关系联动；行为模式偏「师父主导、徒弟配合」。 |
| **家庭/家族** | 家庭单元或家族内小圈子。 | 血缘/婚姻驱动；可与家族势力区分（家族为阵营，此为具体几人）。 |
| **任务小组** | 为完成某项明确任务而组建的团队。 | 目标驱动；任务结束可变更状态；适合限时剧情。 |
| **敌对组合** | 作为对立面一起出现的几人（如反派小组、对手团队）。 | 与主角组对照；可单独维护其集体行为与目标。 |
| **秘密结社** | 地下/秘密小团体，成员身份可能部分对外隐藏。 | 信息差、身份揭露等剧情；可选「仅部分成员公开」。 |
| **其他** | 自定义或上述未覆盖的类型。 | 由创作者自行约定含义。 |

**实现建议**：主表增加 `group_type` 字段，取值可为枚举（如 `fixed_team` / `ad_hoc` / `meeting` / `master_disciple` / `family` / `task_force` / `antagonist_group` / `secret_society` / `other`）或短字符串，便于前端展示与筛选。可选增加 `group_type_notes` 用于对「其他」类型的补充说明。

### 6.2 角色组状态（group_status）

状态描述该组在**当前剧情时间点**的生命周期与是否「可用」，便于随时间线推进调整（如任务结束、小队解散、暂时休眠）。

| 状态枚举建议 | 说明 | 剧情用途 |
|--------------|------|----------|
| **活跃** | 当前正在作为整体活动，可直接用于选角与生成。 | 选角列表默认展示；生成时按「当前有效组」参与上下文。 |
| **休眠** | 暂时不一起行动，但设定上仍存在，后续可再次集结。 | 选角可过滤或弱化展示；生成时可提示「该组当前休眠，若剧情需要可临时启用」。 |
| **已解散** | 设定上该组合已不再存在（如任务结束、分道扬镳）。 | 选角通常隐藏或单独筛「历史组」；生成时可不带入，或仅作背景说明。 |
| **临时集结** | 本为临时组合，当前正处于「集结中」的剧情阶段。 | 与「临时组合」类型配合；可设集结起止时间或关联事件。 |
| **分裂中** | 组内出现分歧，正在走向分裂或重组。 | 过渡态；生成时可强调内部张力、意见不一。 |
| **重组中** | 成员或目标在调整，尚未稳定为新组。 | 过渡态；可与新组、旧组做关联（可选后续扩展）。 |

**实现建议**：主表增加 `group_status` 字段，取值枚举如 `active` / `dormant` / `dissolved` / `ad_hoc_assembled` / `splitting` / `reforming`。可选增加 `status_since`（状态变更时间或剧情时间点）、`status_notes`（如解散原因、休眠说明），便于与时间线或章节挂钩。

### 6.3 类型与状态的组合使用

- **选角/生成**：默认仅展示「活跃」且类型匹配当前场景的角色组；高级筛选项可包含「休眠」「已解散」等，用于回忆杀、闪回或「昔日小队再聚」等剧情。
- **时间线/事件**：若事件与角色组绑定，可约定事件发生时该组的状态（如「某任务结束 → 任务小组置为已解散」）；类型为「临时组合」「任务小组」时，状态流转更频繁。
- **提示词**：生成时可将 `group_type`、`group_status` 写入上下文，例如：「当前角色组类型：临时组合；状态：临时集结。请按短期合作、尚未形成固定默契来描写互动。」

### 6.4 数据模型补充

类型与状态对应字段已纳入 **第四节 基础模型** 与 **5.3 扩展数据模型汇总** 中的主表 `role_group`：`group_type`、`group_status` 建议在 Phase 1 即纳入；`group_type_notes`、`status_since`、`status_notes` 为扩展字段，可随 Phase 3 或时间线联动再完善。

## 七、与现有模块的集成

- **角色管理**：成员从当前世界观下的角色/role_info 中选择，不重复存角色详情，仅存引用。
- **章节/脑洞**：
  - 若传入「角色组」ID 或名称，可增加 `role_group_names` 或在下游将角色组展开为：组内 `role_names` + 一段「集体行动说明」及扩展字段拼入 `curr_context` 或 attention。
  - 选角/上下文 UI 可增加「角色组」多选，默认按 **类型**、**状态** 筛选（如仅展示「活跃」组）；选中后自动带出成员与集体设定（含扩展字段中的行动模式、风格、目标、禁忌等）。可将 `group_type`、`group_status` 一并写入提示词以增强剧情灵活性。
- **Pick 能力**：在 chapters/pick 中增加 target 类型 `role_groups`（可选），从文本中抽取「角色组」名称，与现有 `roles`/`factions`/`locations` 并列。

## 八、非功能需求

- **权限与导航**：与现有角色/阵营一致，依赖权限树配置；在 router 与 navigator 中增加「角色组」入口（如 `novel/roleGroupManage`）。
- **性能**：列表/树按 worldview 过滤，成员数量建议限制（如单组 ≤20），避免大列表拖垮 UI。
- **一致性**：命名与现有 aiNoval 拼写一致，API 路径建议 `/api/aiNoval/roleGroup/*` 或 `/api/aiNoval/characterGroup/*`。

---

## 九、技术架构要点

- **前端**：新增 `src/business/aiNoval/roleGroupManage/`（或 `characterGroupManage`），参考 roleManage、factionManage 的结构：Context、列表/树、编辑弹窗、API 封装。
- **后端**：新增 Service（如 RoleGroupService）、API 路由（如 `pages/api/aiNoval/roleGroup/*`、`pages/api/web/aiNoval/roleGroup/*`），以及 MySQL 表结构。
- **类型**：在 `IAiNoval.ts` 中增加 `IRoleGroup`、`IRoleGroupMember`（含扩展字段的可选属性）。
- **章节集成**：在 chapterManage 的 apiCalls 与选角 UI 中增加「角色组」的入参与展示逻辑；若做 pick，则扩展 pick 的 target 与提示词。

---

## 十、实施阶段与优先级

1. **Phase 1**：数据模型与基础 CRUD（表结构、Service、API、类型定义）+ 角色组管理页（列表、新建/编辑/删除、成员选择）。主表含基础字段及 **角色组类型（group_type）**、**角色组状态（group_status）**，便于首版即支持按类型/状态筛选与剧情灵活性。
2. **Phase 2**：章节侧集成（选角处可选角色组、按类型/状态筛选、将角色组展开为成员 + 集体描述参与生成）。
3. **Phase 3**：扩展字段落地（小组内交互、自然小组行动、类型/状态备注与时间等），并在管理页与章节上下文中展示、写入提示词。
4. **Phase 4**（可选）：Pick 支持 `role_groups`；脑洞分析中引用角色组；时间线/事件与角色组状态联动（如任务结束自动置为已解散）。
